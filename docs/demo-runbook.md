# INFERS デモ運用ランブック (v1.0)

MT5 **デモ口座** での無人稼働(設計書フェーズ8)の運用手順。本書は特に
**G3 ニュース遮断が未実装の間の「手動ニュース回避」運用**(暫定運用)を定める。
G3 の自動実装が完了したら本書の手動手順は不要になる(将来課題: フェーズ8並行課題)。

> **大前提(設計書 §11 / CLAUDE.md 安全原則)**
> - 必ず **デモ口座** から始める。実口座は `--allow-real-account` の明示が必要で、
>   デモ検証完了まで使わない。
> - **防御(損切り・建値SL・半分利確・キルスイッチ)は LLM 非依存で常時動作する。**
>   AI ゲートが全停止しても新規が止まるだけで、保有玉の防御は動き続ける。
> - 確定ベースラインは `rule_depth50`(`depth_max=0.50`)。ゲートは既定のルール
>   判定($0)で運用する(詳細: [reports/README.md](../reports/README.md))。

---

## 1. 起動(v1.0 構成)

PowerShell で venv を有効化してから:

```powershell
python -m infers.main --mode live --demo --symbol XAUUSD `
  --macro-wave2 --depth-screen --depth-max 0.50 --no-fib-score
```

- `--demo` は既定。判断は追記専用ジャーナル
  `work/journal/XAUUSD_<UTC日付>.jsonl` に自動で永続化される(`--journal PATH` で変更可)。
- ゲートは既定で `--ai-client rule`($0・決定論)。LLM ゲートを使う場合のみ
  `--ai-client claude` と `ANTHROPIC_API_KEY` 環境変数が必要。
- 口座資格情報・API キーは **環境変数のみ**(コード・設定・ログに書かない)。

> **ウォームアップの注意**: ライブはコールド起動で、上位足インジケーター
> (D1 の 200SMA/ダウ等)が揃うまで新規シグナルは出ない。デモ開始直後に
> シグナルが出なくても異常ではない。十分な履歴が貯まるまで待つ
> (事前ウォームアップの自動化は将来課題)。

## 2. 手動ニュース回避(G3 未実装の間の暫定運用)★本書の中核

**なぜ必要か**: 重要経済指標(米雇用統計 / CPI / FOMC 等)の発表前後は
スプレッド拡大・価格スパイクが激しく、手法が想定しない損失・スリッページが
起きやすい。手法ゲート3は「重要指標 **±30分** は全面見送り」と定めるが、
自動の経済カレンダー連携(G3)はまだ無いため、**オペレーターが手動で代替**する。

> 自動の安全網として `max_spread_ticks`(`SPREAD_ABNORMAL`)が、スプレッドが
> 開いた瞬間の新規を自動拒否する。ただしこれは **事後的な代理** であり
> G3 の代替にはならない(下記の手動停止を省略しないこと)。

### 毎営業日の手順

1. **当日の禁止ウィンドウを洗い出す**
   経済カレンダーで、当日の **USD / 金(XAU)関連の高インパクト指標** の発表時刻を
   UTC で確認する(雇用統計・CPI・PPI・FOMC・パウエル発言・小売売上高 等)。
2. **各指標に ±30 分の禁止ウィンドウを設定**
   「発表 **30 分前** 〜 発表 **30 分後**」を新規エントリー禁止帯とする。
3. **ウィンドウ開始前に安全停止**
   発表 30 分前までに、稼働中のプロセスを **`Ctrl+C` で停止**する。
   停止時に `shutdown()` が走り、**未約定の打診指値を取消し**、残玉を手仕舞う
   (停止後にブローカーへ孤児注文を残さない)。
4. **ウィンドウ終了まで停止維持**
   発表 30 分後まで再起動しない(相場が落ち着くまで待つ)。
5. **ウィンドウ後に再起動**
   手順 1 のコマンドで再起動する。起動時リコンサイルがブローカー実態と
   ローカル状態を同期する。ジャーナルは同じ日付ファイルへ **追記** される
   (過去ログは消えない)。

### 当日の運用シート(記入テンプレ)

| 指標 | 発表(UTC) | 停止(−30分) | 再開(+30分) | 実施 |
|---|---|---|---|---|
| 例: US CPI | 12:30 | 12:00 | 13:00 | ☐ |
|  |  |  |  | ☐ |

> **既知の割り切り**: 現状の安全停止は **保有玉も手仕舞う**(保守側)。
> 含み益のランナーを早めに閉じる機会損失はあるが、デモ(検証目的)では
> 単純・安全を優先する。「新規だけ止めて保有玉は SL で維持する」細粒度の
> 一時停止は、G3 自動実装(フェーズ8並行課題 #7)で対応する。

## 3. 日次監視(ジャーナル点検)

1 日 1 回、当日のジャーナルを点検する:

```powershell
python -m infers.journal replay --file work/journal/XAUUSD_<UTC日付>.jsonl
```

確認ポイント:
- `kinds:` の `VERDICT` / `FSM` 件数 — 判断と執行が記録されているか。
- `positions:` の遷移タイムライン — 打診→約定→建値SL→利確→決済が想定どおりか。
- **`RECONCILE` イベントで `ok=false` が無いか**(下記「異常時」へ)。
- ルールゲート運用なら末尾の `golden-replay: ... OK`(記録判断が現行ルールで
  再現される=退行なし)。

## 4. 異常時対応

| 事象 | 症状 | 対応 |
|---|---|---|
| **キルスイッチ作動** | 新規が全拒否される(`KILL_SWITCH` 理由)。リコンサイル不一致・日次損失上限が原因 | ラッチ式で自動解除しない。**原因を確認**(ジャーナル/ブローカー実態)し、解消後に**プロセス再起動**で解除。 |
| **リコンサイル不一致** | ジャーナルに `RECONCILE ok=false`(orphan = ローカルが知らない実態ポジション、または vanished/数量不一致) | 自動補正せずキルスイッチで新規停止。ブローカー画面で実態を確認し、身に覚えのないポジション/注文を手動処理してから再起動。 |
| **フィード回復不能** | プロセスが `FeedError`(再接続上限超過)で終了 | MT5 ターミナルの起動・ログイン・ネットワークを確認し、再起動。起動時リコンサイルで同期される。 |
| **日次損失上限到達** | キルスイッチ作動 | その日は再開しない。翌営業日(日次カウンタは UTC 確定足境界でリセット)に原因を確認のうえ再起動。 |

> いずれの異常でも **保有玉の SL は発注と同時に設定済み** であり、プロセスが
> 落ちてもブローカー側の SL が防御を継続する(SL なしの保有状態は構造的に存在しない)。

---

## 5. 停止・終了

- **一時停止(ニュース回避など)**: `Ctrl+C` → 安全停止(未約定取消+手仕舞い)→ 後で再起動。
- **完全終了**: 同じく `Ctrl+C`。`finally` でフィードとジャーナルが閉じられる。

## 6. チェックリスト(デモ開始前)

- ☐ MT5 デモ口座にログイン済み・ターミナル起動済み
- ☐ 口座資格情報/API キーは環境変数で設定(ファイルに書いていない)
- ☐ 起動コマンドが v1.0 構成(`--macro-wave2 --depth-screen --depth-max 0.50 --no-fib-score`)
- ☐ 当日の経済指標カレンダーを確認し禁止ウィンドウを運用シートに記入
- ☐ `work/journal/` の出力先が書き込み可能
- ☐ 最小ロット設定(`DEFAULT_RISK` / `FsmConfig` を口座規模に対し保守的に)
